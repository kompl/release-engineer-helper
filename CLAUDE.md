# CLAUDE.md — release-engineer-helper

## Проект

Go-утилита для анализа падений CI-тестов в GitHub Actions. Собирает историю запусков, классифицирует упавшие тесты (stable_failing, flaky, fixed, single_failure), строит HTML/JSON-отчёты.

## Триггер анализа отчёта

Когда пользователь скидывает ссылку на JSON-отчёт (путь к файлу или `@file` reference) — это запрос на **исследование текущей CI-ситуации**. Не нужно ждать дополнительных инструкций. Сразу прочитать файл и выдать анализ.

## Как исследовать JSON-отчёт

### 1. Структура отчёта

Отчёт содержит `projects` — map по ключу `{repo}/{branch}`. Каждый проект включает:
- `latest_run` — последний CI-запуск с деталями упавших тестов
- `summary` — агрегированная статистика по всем проанализированным ранам

### 2. Порядок анализа

1. **Общая сводка** — таблица по всем веткам: кол-во ранов, уникальных падений, stable_failing, flaky, fixed. Это даёт мгновенное понимание масштаба проблем.

2. **Стабильно падающие тесты (stable_failing)** — главный приоритет. Это реальные регрессии:
   - Смотреть `probable_cause` — какой коммит начал серию падений, длина серии (`streak_length`)
   - Смотреть `failing_since` — как давно падает
   - Смотреть `pattern` — должен быть сплошной блок 🔴 в конце
   - Смотреть `error_message` — тип ошибки (ORA-*, timeout, assertion, etc.)

3. **Flaky тесты** — оценить масштаб проблемы:
   - `fail_rate_pct` — чем выше, тем больше влияние на CI
   - `pattern` — хаотичное чередование 🔴🟢 подтверждает flakiness
   - `flaky_info.fail_count / total_runs` — абсолютные числа
   - Сгруппировать по типу ошибки (timeout-ы обычно имеют общую причину)

4. **Динамика и тренды**:
   - Сравнить ветки между собой — какая здоровее, какая деградирует
   - `fixed_count > 0` — положительный сигнал, кто-то чинит
   - `new_failures` vs `total_runs_analyzed` — скорость появления новых проблем
   - Молодые ветки (мало ранов) с большим числом падений — тревожный сигнал

5. **Кросс-ветковые паттерны**:
   - Одинаковые тесты падают в нескольких ветках → системная проблема
   - Тест падает только в одной ветке → специфика ветки / конкретного коммита

### 3. Классификации тестов

| classification | Значение | Приоритет |
|---|---|---|
| stable_failing | Падает стабильно от определённого коммита | Высокий — регрессия |
| flaky | Хаотично то падает, то проходит | Средний — шум в CI |
| fixed | Раньше падал, сейчас проходит | Информационный |
| single_failure | Упал один раз | Низкий |

### 4. Формат ответа

- Начать с таблицы-сводки по веткам
- Выделить stable_failing как приоритетные проблемы
- Сгруппировать flaky по типу ошибки (timeouts отдельно, ORA-ошибки отдельно, etc.)
- Отметить кросс-ветковые паттерны
- Дать оценку динамики: улучшается / ухудшается / стабильно
- Если есть `probable_cause` — указать коммит и PR как вероятный источник
